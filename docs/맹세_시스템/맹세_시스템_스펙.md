# 맹세(盟誓) 시스템 스펙

> 플레이어가 스스로 자신을 약화시키는 자기 제약 시스템.
> 약화의 무게에 비례해 숙련도·드랍률을 부스트한다.
>
> 본 문서는 시스템 차원의 룰만 정의한다. 개별 맹세 옵션의 수치·이름은
> `맹세_옵션_카탈로그.md` 참조.

---

## 1. 핵심 정의

- **맹세(盟誓)**: 플레이어가 전투 진입 전 스스로 부과하는 약화 효과.
- **율법(律法) / 철칙(鐵則)**: 적이 가진 패시브. 플레이어 빌드를 보고 적이 받는 피해가 변동.

| 축 | 발동 주체 | 효과 방향 | 보상 가중치 |
|---|---|---|---|
| 율법·철칙 | 적 | 적이 받는 피해 감소 | **카운트하지 않음** |
| 맹세 | 플레이어 | 플레이어 능력치·자원 약화 | **이것만 카운트** |

두 축은 **별개로 곱연산** 적용된다. 율법이 발동했더라도 그것이 맹세는 아니므로 보상 부스트에 영향이 없다.

## 3. 데이터 모델

### 3-1. 맹세 정의 (`data/oaths.ts` 신규)

```ts
export type OathCategory = 'maxQi' | 'recovery' | 'output' | 'incoming' | 'future';

export interface OathEffect {
  // 자원
  maxQiPenaltyPct?: number;          // 0..1, maxQi 곱연산 감소
  hpRegenPenaltyPct?: number;        // 0..1, hp regen 곱연산 감소
  hpDrainPctPerSec?: number;         // maxHp 대비 매초 감소 비율 (0..1)

  // 데미지
  outDamagePenaltyPct?: number;      // 0..1, 출력 데미지 곱연산 감소
  inDamageBonusPct?: number;         // 0..1, 받는 피해 곱연산 증가

  // 미래 확장
  monsterSkillVariantTrigger?: string;  // 적 스킬 분기 트리거 키
}

export interface OathDef {
  id: string;
  name: string;
  flavor: string;                  // 이탤릭 무협체 한 줄
  description: string;             // 메커니즘 한 줄
  category: OathCategory;
  weight: number;                  // 1~5
  effect: OathEffect;
  exclusiveGroup?: string;         // 같은 그룹 내 1개만 활성 (단계 옵션 그룹화)
}

export const OATHS: Record<string, OathDef> = {
  // 카탈로그 문서 참조
};
```

## 4. 해금·잠금 정책

### 4-1. 필드별 해금 (boss-kill gate)

각 필드에서 맹세 시스템이 활성되려면 **해당 필드의 보스를 처치**해야 한다.

```
isOathEnabledInField(fieldId, state) =
  FieldDef.boss != null && state.bossKillCounts[FieldDef.boss] > 0
```

- **보스가 없는 필드** (수련장, 보스 미정 답파 필드 등): **영구 사용 불가**
- **보스가 있는 필드**: 해당 보스 처치 후 활성

이 룰은 단일하다 — 외문이든 야산이든 동일. 새 필드 추가 시 별도 정책 없이 자동 적용.

### 4-2. 미해금 필드 진입 시 동작 (Option B)

| 항목 | 동작 |
|------|------|
| 활성 토글 (`activeOathIds`) | **유지** — 마을 복귀 또는 해금된 필드 진입 시 그대로 적용됨 |
| 잠금 (`lockedAt`) | **박제하지 않음** — null 유지 |
| 효과 (페널티) | 적용 안 함 |
| 보상 부스트 | 적용 안 함 |
| UI | "이 필드는 [보스명] 처치 후 사용 가능" 안내 배너 |

유저가 마을에서 맹세를 세팅해두면 해금된 필드만 자동 발동하고, 미해금 필드는 평범한 전투로 진행. 매번 토글할 필요 없음.

### 4-3. 해금된 필드 진입·이탈

| 상태 | 토글 | lockedAt | 효과·보상 |
|------|------|----------|-----------|
| 마을 (`currentField == null`) | 자유 | null | 미적용 |
| 해금된 필드 진입 시 | **잠금** | `{ fieldId, lockedAtTimestamp, snapshotIds }` | 적용 |
| 해금된 필드 이탈 (마을 복귀·필드 이동) | 잠금 해제 | null | — |

**효과 적용의 단일 게이트**: `state.oathSystem.lockedAt != null` — 이것이 true일 때만 모든 페널티·보상 부스트 적용. 미해금 필드에선 lockedAt이 박제되지 않으므로 자동으로 단락(short-circuit).

**토글 게이트**: `state.currentField == null` — 마을에서만 토글 가능. 어떤 필드든 진입 후엔 토글 비활성 (해금/미해금 무관, UI 일관성).

### 4-4. 적·필드 차단 (forbidOathIds)

`forbidOathIds`에 포함된 맹세는 해당 필드/적 전투에서 효과·보상 모두 적용되지 않음. weightSum 계산에서도 제외. UI에서 회색 처리 + 툴팁 표시. 4-1의 해금 게이트와는 별도 레이어.

## 5. 합산 규칙 (4번 결정 디테일)

데미지 계산 파이프라인은 **두 페널티 레이어를 모두 곱연산**으로 적용한다.

### 5-1. 플레이어가 적에게 가하는 데미지

```
finalDamage = baseDamage
  × (1 - oathOutDamagePenalty)        // 맹세 (플레이어 측)
  × (1 - enemyDamageReductionByLaw)   // 율법·철칙 (적 측)
  × (기타 곱연산)
```

### 5-2. 플레이어가 받는 데미지

```
finalIncoming = enemyAttackBase
  × (1 - playerDamageReduction)       // 외법·심득 등 기존
  × (1 + oathInDamageBonus)           // 맹세 (플레이어 측)
```

### 5-3. 보상 산정 (계단형 piecewise)

```
weightSum = sum(activeOathIds.map(id => OATHS[id].weight))
```

가중치 합에 따라 4개 티어로 나뉘며, **각 가중치 1단위마다 자기가 속한 티어 비율로 가산**된다 (단순 누적이 아닌 piecewise 적분 형태).

| 티어 | weightSum 범위 | 가중치당 가산 | 추가 보너스 (전체 답파 적용) | 도전 등급 |
|------|----------------|----------------|---------------------------------|-----------|
| 1 | 0–4 | 숙련도 +10% / 드랍률 +10% | — | 적당한 도전 |
| 2 | 5–9 | 숙련도 +20% / 드랍률 +20% | 일부 몬스터 드랍 테이블 추가 (대상 추후 지정) | 힘겨운 도전 |
| 3 | 10–17 | 숙련도 +30% / 드랍률 +30% | 모든 몬스터 숙련도 등급 **+1** | 무모한 도전 |
| 4 | 18+ | 숙련도 +50% / 드랍률 +50% | 모든 몬스터 숙련도 등급 **+2** (티어 3 대체) | 불가능한 도전 |

**보상 부스트는 답파 중 만나는 모든 몬스터에 적용**된다 (맹세의 효과(약화)는 모든 전투에 적용, 보상 부스트도 동일).

#### 5-3-1. 부스트 함수 (의사코드)

```ts
function calcOathBoost(weightSum: number): { profMult: number; dropMult: number } {
  let boost = 0;
  let remaining = weightSum;
  const tier1 = Math.min(remaining, 4);   boost += tier1 * 0.10; remaining -= tier1;
  const tier2 = Math.min(remaining, 5);   boost += tier2 * 0.20; remaining -= tier2;
  const tier3 = Math.min(remaining, 8);   boost += tier3 * 0.30; remaining -= tier3;
  /* tier4 */                              boost += remaining * 0.50;
  const capped = Math.min(boost, 10.0);    // cap +1000%
  return { profMult: 1 + capped, dropMult: 1 + capped };
}

function calcOathFlatBonuses(weightSum: number): {
  monsterRankBonus: number;
  extraDropTableUnlocked: boolean;
} {
  if (weightSum >= 18) return { monsterRankBonus: 2, extraDropTableUnlocked: true };
  if (weightSum >= 10) return { monsterRankBonus: 1, extraDropTableUnlocked: true };
  if (weightSum >= 5)  return { monsterRankBonus: 0, extraDropTableUnlocked: true };
  return { monsterRankBonus: 0, extraDropTableUnlocked: false };
}
```

율법/철칙은 보상 식에 들어가지 않는다.

## 6. 보상 캡

| 항목 | 캡 | 비고 |
|------|----|------|
| 숙련도 부스트 | **+1000%** | 11배 |
| 드랍률 부스트 | **+1000%** | 11배 |

**캡 도달 가중치**: weightSum ≈ 30 (티어 4 진입 후 13단위 추가).

**MVP 최대 조합** (각 카테고리 최강 단계: 6+4+8+7 = **weightSum 25**) → +780% 부스트. **캡 미도달**. 캡 의미는 v2 확장(B/C/D/E 카테고리) 후 활성화.

전용 재화 ("맹세의 결정" 등)는 본 스펙에서는 **파이프라인 자리만 비워두고** 추후 도입.

## 7. UI 진입점 — 별도 "맹세" 탭

기존 탭 구조에 **맹세** 탭 추가.

탭 구성:
1. **상단 요약 패널** — 현재 활성 맹세 수 / 가중치 합 / 부스트 미리보기 (숙련도 +X% · 드랍률 +Y%)
2. **카테고리 섹션** — 자원 / 회복 / 출력 / 받는 피해 (옵션 카탈로그 기준)
3. **각 옵션 카드** — 이름 · flavor · 효과 수치 · 가중치 · 토글 버튼
4. **잠금 표시** — 필드 진입 후엔 토글 비활성, "현재 N 필드에서 잠김" 배너
5. **차단 표시** — `forbidOathIds`로 막힌 옵션은 회색 + "(이 필드/적은 [맹세명] 무효)" 툴팁

같은 `exclusiveGroup` 내에선 한 단계만 활성 (단계 토글 라디오 형태).

## 8. forbidOathIds 정책

기본은 빈 배열. 적용 예시:
- **외문수좌 보스 전투**: 3종 무공 장착이 핵심 메커니즘이므로 무공 봉인 계열 맹세(미래 B/C 카테고리)를 forbid 예정. 현 MVP의 F 카테고리는 영향 없음.
- 기타 일반 외문 적: 차단 없음. 자유롭게 맹세 적용 가능.

## 9. 율법/철칙과의 관계 (재확인)

- 율법·철칙은 **그대로 유지**. 맹세는 별도 시스템.
- 외문에서 배화교 무공 미장착 + 자기 제약 동시 사용 시:
  - 율법 -50%(또는 철칙 -75%) 발동
  - 동시에 맹세에 따른 페널티 발동
  - 보상은 맹세 가중치만 카운트
- "외문수좌 3종 무공 장착 시 철칙 0% 해제"는 맹세와 무관하게 별도 룰 유지.

## 12. 메모

- `firstEntryEvent` 구조와 통합할지 여부는 차후 결정. 진입 모달이 아니라 별도 탭이지만, "처음 도전 시 안내 로그"는 firstEntryEvent로 처리 가능.
- 율법/철칙 발동 로그와 맹세 발동 로그는 시각적으로 구분 (배너 색·아이콘 다르게).
- 외문 소급 적용은 **외문수좌 처치 후** 활성화 정책 (외문 첫 클리어는 정직하게).
