# 무림 방치록 — 서브에이전트 구조 개선 지시서 (최종)

---

## 핵심 전제

**게임 시스템이 큰 변화를 겪어도 에이전트가 망가지지 않는다.**

이를 위해 에이전트의 의존 범위를 계층화한다:

| 계층 | 예시 | 에이전트 의존 |
|------|------|-------------|
| 구조 | 전투가 있다, 성장이 있다, 몹이 있다 | 에이전트 지시(md)가 의존 |
| 콘텐츠 | 스탯 이름, 몹 목록, 무공 종류 | testAdapter + 데이터 파일 읽기 허용 |
| 수치/공식 | 데미지 공식, 비용 곡선, 패시브 효과 | 의존 금지. 게임을 돌려서 관찰만 |

**에이전트 지시(md)에는 현재 게임의 구체적 구현 상세를 적지 않는다.**
구현 상세는 GAME_GUIDE.md에만 두고, 에이전트 md는 GAME_GUIDE.md를 참조한다.
시스템이 바뀌면 GAME_GUIDE.md만 갱신하면 되고, 에이전트 md는 안 건드린다.

모든 에이전트는 "지시를 읽고 → 현재 코드/파일을 읽고 → 작업한다".
지시에 구현 상세가 없으므로, 구현이 바뀌어도 지시는 유효하다.

## 개선 1: 게임을 테스트 가능하게 만드는 최소 변경

### tick()에 dt 외부 주입 (combat-dev 작업)

현재 tick()은 Date.now()로 dt를 계산한다.
테스트 환경에서 tick()을 빠르게 반복 호출하면
dt가 0에 가까워져서 아무 일도 안 일어난다.

**변경**: tick()이 선택적으로 외부 dt를 받을 수 있게 한다.

```ts
// 변경 전
tick: () => {
  set(state => {
    const now = Date.now();
    const rawDt = (now - state.lastTickTime) / 1000;
    const dt = Math.min(rawDt, 5);
    if (dt < 0.05) return { lastTickTime: now };
    // ...
  });
}

// 변경 후
tick: (forceDt?: number) => {
  set(state => {
    let dt: number;
    let now: number;
    if (forceDt !== undefined) {
      dt = forceDt;
      now = state.lastTickTime + forceDt * 1000;
    } else {
      now = Date.now();
      const rawDt = (now - state.lastTickTime) / 1000;
      dt = Math.min(rawDt, 5);
      if (dt < 0.05) return { lastTickTime: now };
    }
    // 이후 로직 동일 — 이 dt와 now를 사용.
    // 중요: 함수 끝의 return에 lastTickTime: now 가 반드시 포함되어야 한다.
    // (현재 코드 1018줄에 이미 있음. 삭제하지 말 것.)
  });
}
```

이 변경은:
- 기존 게임 동작에 영향 없음 (forceDt 미전달 시 기존과 동일)
- 시스템이 어떻게 바뀌든 "tick에 dt를 넣을 수 있다"는 것만 유지하면 됨
- 한번 해놓으면 영구적으로 유효

### localStorage 조건부 처리 (combat-dev 작업)

gameStore의 saveGame/loadGame에서 localStorage 접근을 조건부로 변경:

```ts
if (typeof window !== 'undefined' && window.localStorage) {
  localStorage.setItem('murim_save', JSON.stringify(saveData));
}
```

이것도 한번 해놓으면 영구적으로 유효.

## 개선 2: 테스트 어댑터

### 역할

게임 상태를 읽고, 행동을 지시하고, 시간을 진행하는 최소한의 레이어.
gameStore를 직접 import하는 유일한 테스트용 접점.
게임 액션을 개별 래핑하지 않고, 범용 호출 함수를 제공한다.
→ 게임에 액션이 추가/삭제되어도 testAdapter를 수정할 필요가 없다.

### `app/src/testAdapter.ts` (combat-dev 작업)

```ts
import { useGameStore } from './store/gameStore';

// === 상태 읽기 ===
// gameStore의 현재 상태를 그대로 반환한다.
// 추상화하지 않는다. 유지보수 지점을 하나로 유지하기 위함.
export function getState() {
  return useGameStore.getState();
}

// === 상태 주입 ===
// 특정 빌드 상태를 직접 세팅할 때 사용한다.
// 예: 경맥합 30 상태에서 곰을 테스트하고 싶을 때,
// 0부터 성장시키는 대신 바로 원하는 상태를 주입한다.
//
// ⚠️ 주의: 게임이 정상적으로 도달 가능한 상태를 세팅할 것.
// 불가능한 조합(예: 경지 3인데 무공 0개, 경맥합 100인데 내공 소비 0)은
// 게임 내부의 불변식을 깨뜨려 테스트 결과를 왜곡한다.
// 확실하지 않으면 resetGame() 후 advanceTime으로 성장시키는 방법이 안전하다.
export function setState(partial: Record<string, any>) {
  useGameStore.setState(partial);
}

// === 액션 호출 ===
// gameStore의 어떤 액션이든 이름으로 호출할 수 있다.
// 게임에 액션이 추가/삭제되어도 이 함수를 수정할 필요가 없다.
//
// 사용 예:
//   callAction('investStat', 'sungi')
//   callAction('startHunt', 'yasan', 'squirrel')
//   callAction('equipArt', 'samjae_sword')
//   callAction('abandonBattle')
export function callAction(name: string, ...args: any[]): any {
  const store = useGameStore.getState();
  const action = (store as any)[name];
  if (typeof action === 'function') {
    return action(...args);
  }
  throw new Error(`Action "${name}" not found in gameStore`);
}

// === 리셋 (편의 함수) ===
// 자주 사용되므로 별도 제공.
export function resetGame() {
  useGameStore.getState().resetGame();
}

// === 시간 진행 ===
// tick(forceDt)를 반복 호출하여 게임을 빠르게 진행한다.
// 매 반복마다 getState()를 새로 호출하여 최신 상태의 tick을 사용한다.
export function advanceTime(seconds: number) {
  for (let i = 0; i < seconds; i++) {
    useGameStore.getState().tick(1.0);
  }
}

// 매 초마다 상태를 확인하며 시간 진행.
// onTick이 true를 반환하면 중단. 실제 진행한 초를 반환.
export function advanceTimeWithCheck(
  seconds: number,
  onTick: (state: ReturnType<typeof getState>, elapsed: number) => boolean
): number {
  for (let i = 0; i < seconds; i++) {
    useGameStore.getState().tick(1.0);
    if (onTick(useGameStore.getState(), i + 1)) {
      return i + 1;
    }
  }
  return seconds;
}
```

### 왜 이 구조인가

- **getState()가 상태를 그대로 반환**: 추상화 레이어 없음. 유지보수 지점 하나.
- **callAction()으로 모든 액션 호출**: 액션 래퍼를 개별로 만들지 않으므로, 게임에 액션이 추가/삭제되어도 testAdapter를 수정할 필요 없음. Zustand 액션은 클로저로 set/get을 캡처하므로 this 바인딩이 불필요하여 `action(...args)`로 직접 호출한다.
- **setState()로 상태 주입 가능**: 원하는 빌드를 바로 세팅.
- **advanceTime이 매번 getState()를 호출**: Zustand 상태 갱신이 정확히 반영됨.
- **게임 시스템이 바뀌면**: getState/setState/callAction/advanceTime 모두 안 깨짐. 테스트 스크립트만 새로 짜면 됨.

### 실행 환경

테스트 스크립트: `app/scripts/test-[이름].ts`
실행: `cd app && npx tsx scripts/test-[이름].ts`
Zustand는 Node.js에서 React 없이 동작한다 (vanilla store).

## 개선 3: 기존 에이전트 지시 개선

### 원칙
에이전트 md에서 현재 게임의 구현 상세를 제거하고 GAME_GUIDE.md로 위임한다.
에이전트 md에는 "역할", "작업 방식", "금지 사항"만 남긴다.

### 모든 에이전트 공통: GAME_GUIDE.md 갱신 규칙

모든 에이전트의 "작업 방식"에 아래 규칙을 포함한다:

```
- 자신의 작업이 GAME_GUIDE.md의 기존 내용과 달라지는 변경을 포함할 경우,
  GAME_GUIDE.md의 해당 섹션도 함께 갱신한다.
  갱신 시 무엇을 어느 섹션에 어떻게 바꿨는지 작업 완료 보고에 포함한다.
```

이 규칙에 의해 GAME_GUIDE.md는 항상 최신 상태를 유지한다.
기획자가 직접 갱신할 필요가 없다.
해당 영역의 작업을 한 에이전트가 자기 작업의 일부로 갱신한다.

  "전투 로직 수정해줘", "무공 발동 바꿔줘", "전투 버그 수정해줘" 등의 요청 시 자동 위임.

당신은 무림 방치록의 게임 로직 전문가입니다.

## 반드시 참조
- `GAME_GUIDE.md`를 먼저 읽고, 현재 게임의 전투 원칙과 설계 철학을 파악합니다.
- 작업 전에 관련 코드 파일을 직접 읽어 현재 구현 상태를 확인합니다.

## 담당 범위
- gameStore의 게임 로직 (전투 틱, 성장, 상태 관리)
- 데이터 파일의 구조 정의 (인터페이스)
- `app/src/testAdapter.ts` 유지보수
- tick()의 forceDt 파라미터 유지

## 작업 방식
- 현재 전투 원칙은 GAME_GUIDE.md에 정의되어 있다.
  구현은 이 원칙을 따르되, 구체적 방법은 코드에서 판단한다.
- 자신의 작업이 GAME_GUIDE.md의 기존 내용과 달라지는 변경을 포함할 경우,
  GAME_GUIDE.md의 해당 섹션도 함께 갱신한다.
  갱신 시 무엇을 어느 섹션에 어떻게 바꿨는지 작업 완료 보고에 포함한다.
- 게임 시스템을 변경할 때 testAdapter.ts도 확인한다.
  testAdapter의 getState/setState/callAction/advanceTime은
  게임 액션에 의존하지 않으므로 대부분 갱신 불필요.
  tick()의 시그니처가 바뀌는 경우에만 advanceTime을 맞춘다.
- 작업 완료 후 기획자에게
  "balance-tester로 밸런스 재검증을 권장합니다"라고 안내한다.

## 유지해야 하는 것
- tick()은 항상 외부에서 dt를 주입할 수 있어야 한다:
  `tick(forceDt?: number)` 시그니처를 유지한다.
- gameStore에서 브라우저 전용 API(localStorage 등) 사용 시
  항상 조건부 처리: `if (typeof window !== 'undefined') { ... }`
- 데이터 파일(data/ 디렉토리)은 gameStore나 다른 로직 파일을 import하지 않는다.
  데이터 파일은 순수 데이터와 타입 정의만 포함한다.
  (테스트 스크립트가 데이터 파일을 직접 import하므로, 연쇄 의존이 없어야 한다.)

## 금지
- 몬스터/무공 데이터 수치(HP, 공격력 등)를 직접 변경하지 않는다.
  데이터 수치 조정은 balance-tester가 담당.
- UI 컴포넌트를 직접 변경하지 않는다.

- `GAME_GUIDE.md` (밸런스 원칙, 테스트 체크리스트)
- `app/src/testAdapter.ts` (테스트 인터페이스 — 사용법과 주의사항이 주석에 있음)

## 핵심 원칙

### 게임을 돌려서 관찰한다
전투 공식, 데미지 계산을 직접 재현하지 않는다.
testAdapter로 게임을 실제로 돌리고 결과를 측정한다.
공식이 바뀌어도 테스트 방식은 동일하다: 돌리고, 읽고, 보고한다.

### 데이터 파일은 읽어도 된다
몹 목록, 무공 목록, 전장 구조 등 데이터 파일은 참조할 수 있다.
테스트 스크립트에서 몹 ID, 전장 ID를 사용해야 하므로 필요하다.
단, 데이터 파일의 수치로 데미지를 직접 계산하지는 않는다.

### 판단하지 않는다, 분석한다
밸런스가 좋다/나쁘다를 결정하지 않는다.
관찰한 사실을 데이터로 정리하고, 조정 옵션을 제시한다.
최종 결정은 기획자가 한다.

## 테스트 방법

### 테스트 스크립트 작성
기획자의 질문에 맞는 테스트 스크립트를 작성한다.
스크립트는 `app/scripts/test-[설명].ts`에 둔다.
실행: `cd app && npx tsx scripts/test-[이름].ts`

스크립트는 testAdapter의 함수로 게임을 조작한다:
- `getState()`: 현재 게임 상태 읽기
- `setState({...})`: 특정 빌드 상태 직접 세팅 (주의사항은 testAdapter.ts 주석 참고)
- `callAction('액션명', 인자...)`: 게임 액션 호출
- `advanceTime(초)`: 시간 빠르게 진행
- `advanceTimeWithCheck(초, 콜백)`: 매 초 상태 확인하며 진행
- `resetGame()`: 초기화

### 재사용 가능한 패턴
자주 쓰는 테스트 패턴은 헬퍼 함수로 만들어
`app/scripts/test-helpers.ts`에 모아둔다.
헬퍼는 testAdapter 함수와 데이터 파일만 참조한다.
게임 시스템이 바뀌어서 헬퍼가 깨지면 해당 헬퍼만 고치면 된다.

## 작업 흐름

### "현재 밸런스를 진단해줘"
1. 테스트 스크립트 작성 및 실행
2. 여러 플레이 패턴으로 반복 (효율 추구, 도전형, 균형형)
3. 보고서 제출

### "X를 Y로 바꾸면 어떻게 돼?"
1. 현재 상태로 기준선 데이터 수집
2. 변경이 데이터 수치면: 임시 변경 → 재테스트 → 비교
3. 변경이 로직이면: "combat-dev에게 의뢰 필요" 안내
4. 변경 전후 비교 보고서 제출

### "이 수치를 바꿔줘"
1. 지시된 수치 변경 적용 (데이터 파일만)
2. 변경 전후 비교 테스트
3. 결과 보고서 제출

## 개선 5: GAME_GUIDE.md 마일스톤 섹션

GAME_GUIDE.md 5장에 추가:

```markdown
### 마일스톤 측정 항목

balance-tester는 진행도 테스트 시 아래 항목의 도달 시간을 측정하고 보고한다.
"도달"의 구체적 기준은 테스터가 정의하되, 보고서에 명시한다.

**전장별 공통 마일스톤:**
- 전장 진입: 해당 전장의 몹을 처음 처치
- 안정 사냥: 가장 약한 몹을 반복 사냥해도 사망하지 않는 시점
- 중반 진입: 전장 중간 난이도 몹을 안정 사냥
- 최강 도달: 전장 최강 일반 몹을 안정 사냥
- 보스 클리어: 해당 전장 답파 성공
- 경지 돌파: 다음 경지 조건 충족

**측정 시 포함해야 하는 정보:**
- 도달 시점의 플레이 시간 (분)
- 도달 시점의 빌드 상태 (스탯, 무공, 경지)
- 이전 마일스톤으로부터 걸린 시간
- 해당 구간에서 주로 사냥한 몹

**건강한 밸런스의 징후:**
- 마일스톤 간 간격이 점진적으로 늘어난다
- 효율 추구 플레이와 도전 플레이의 최종 격차가 과도하지 않다
- 최적 사냥 대상이 성장에 따라 상위 몹으로 이동한다
- 특정 구간에서 지나치게 오래 정체하지 않는다

이 징후는 절대적 기준이 아니라 방향이다.
실제 밸런스는 테스트 데이터를 보고 기획자가 판단한다.
```

## 부록: 이 구조가 시스템 변경에 강건한 이유

**스탯 시스템을 바꾸면?**
→ gameStore의 stats 구조가 바뀜
→ testAdapter는 안 깨짐 (callAction은 이름으로 호출하므로)
→ 에이전트 md는 안 깨짐 (스탯 이름이 md에 없으므로)
→ combat-dev가 GAME_GUIDE.md의 해당 내용을 갱신
→ 기존 테스트 스크립트 중 stats를 참조하는 것만 깨짐 → 새로 짜면 됨

**전투 시스템을 갈아엎으면?**
→ gameStore의 tick() 내부가 완전히 달라짐
→ testAdapter는 tick(forceDt)를 호출할 뿐이므로 안 깨짐
→ combat-dev.md는 안 깨짐 (전투 메커니즘 상세가 없으므로)
→ combat-dev가 GAME_GUIDE.md의 전투 원칙 섹션을 갱신

**새 액션을 추가하면? (예: doMeditate)**
→ testAdapter 변경 없음 (callAction('meditate')로 바로 호출 가능)
→ 에이전트 md 변경 없음

**내공 시스템을 없애면?**
→ testAdapter는 안 깨짐 (getState는 상태를 그대로 반환)
→ 에이전트 md는 안 깨짐 (내공이라는 단어가 md에 없으므로)
→ 테스트 스크립트만 깨짐 → 새로 짜면 됨

**새 전장을 추가하면?**
→ testAdapter 변경 없음
→ 에이전트 md 변경 없음
→ balance-tester가 새 전장용 테스트 스크립트를 추가로 짜면 됨

**패턴**: testAdapter는 안 깨지고, 에이전트 md는 안 깨지고,
GAME_GUIDE.md는 변경을 한 에이전트가 자동으로 갱신하고,
깨지는 건 테스트 스크립트이며, 테스트 스크립트는 상황에 맞게 새로 짜는 것이다.
